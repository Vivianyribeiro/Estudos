	 - custo = dentro do orçamento (PLANEJAMENTO E CONTROLE)
	- qualidade = se atende aos requisitos de qualidade (PLANEJAMENOT EXECUÇÃO E CONTROLE)
	- recursos humanos - assegura o uso efetivo dos recursos humanos do pjto (PLANEJAMNEOT E EXECUÇÃO)

REQUISITOS E ENFGENHARIA DE REQUISITOS

** REQUISITOS = refletem os objetovos, necessidados ds interessadops e caracteristicas de qualidades a ser atendidas
	REQUISITOS DE SOFTWARE
- GERAIS = que é comum para aquele tipo
- funcionais  = definem parte das funcionalidads (foca no problema)
-IMPLEMENTAÇÃO = restringem a forma como o sistemas deve ser implementado
-DE DESEMPEMNHO = especificam o nivel minimo de desenpenho aceitavel
-DE USABILIDADE = especidficam oq o grupo de usuarios pode ver

** TIPOS DE REQUISITOS:
	- FUNCIONAIS = em gereal define o escopo do sistema, relacionado com as funçoes basicas do sistema(focam no problema)
	- nao funcionais = relacionados com a caracteristicas que qualificam uma solução para o problema(focam na solucao)
	- restrições = são limitações que devem ser consideradas
outros : de sistema , de usuarios vs requisitos de software

** ENGENHARIA DE REQUISITOS 
	conjunto de processos envolvidos com a descobverta (levantameno, coleta), documentaçã e manutençãodos reqisitos. o termo 'engenharia ' implica o uso sistematico e repetido de tecnicas para assegurar que os requisitos estejam completos , consistentes e relevantes

** ENGENHARIA DE REQUISITOS VS ANALISE DE SISTEMAS
 executam as mesmas tarefas porem a engenharia se preucupa mais em relação aos processos adotados, a documentação de requisitos e ao gerenciamento dos requisitos

**PROCESSO DE ENGENHARIA DE REQUISITOS
	
	Conunto estruturado de atividades que permitem deriva, validar e manter os requisitos de um sistema e inclui = elicitação de requisitos, analise de requisitos, negociação de requisitos e validação de requisitos, e gerencia de requisitos

** ATIVIDADES DO PROCESSO DE ENGENHARIA DE REQUISITOS
GERENCIAMENTO DE REQUISITOS ENVOLVE: 
	- ELICITAÇÃO DE REQUISITOS - identifica reqsits  (necessidades de usuarios, informaçoes d dominiio, normas reguladoras padroes)
	- ANALISE E NEGOCIAÇÃO = req analisados e os representantes de cada grupo de usuarios devem negocialos de forma a chegarem em um consenso. Divergencias sao inevitaveis pois a limitações de orçamento prazo imconmpatibiidade e conflitos e perspecticvas diferentes entre usuarios. Checklist para organizar bababa

	- DOCUMENTAÇÃO DE REQUISITOS = É UMA DECLARAÇÃO OFICIAL dos requisitos p/ todos enteressados, sao documentados e formalizados em um doc de requisitos, cujo entendiemnto devera ser comum a todos os envolvidos e o processo de ER nao estara completos enquanto este dipcumento nao estiver sido completado = TEM O PROPOSIOT DE prover meios de comunicação entre clientes usuarios engenheiros de sw e demais para suportar a fase de testes de sistema, para contriolar a evolução do sistema - é importante identificar todos os interessados para que tem todos os requisitos necessarios
	- VALIDAÇÃO D REQUISITOS = identificar possiveis problemas nos requisitos antes que o docuneto de requisistos produzdo seja utilizado para desenvolvimento do sistema - FORMA UM CIRCULO COM A DOCUMENTAÇÃO ATE ESTA COMPLETA sem problemas

gerenciamento de requisitpos tm envolve mudanças controle de modificações analise de impacto etc.

** CAUSAS DE MUDANÇAS - erros nos requisitos, omissoes e novos req identificados
	- sao inevitaveis a medida que usarios desenviolvem uma melhor compreencao de suas necessidades
	- o ambiente tecnico, politico e organizacional pode sofrer mudancas

	& nivel inicial = nao é bem definido, problemas intensos e alto indice de retrabalho

	& intermediario = existem padroes bem definidos para o doc de req, e para descriçoes desses requisitps, sao introduzidas politicas e procedimentos para o gerenciamneto de requisitos

	& MADURO = process muito bem definido e aplicavel para as atividaes de requisitos, alem de haver um program de MELHORIA CONTINUA do proces de ER em andamento
